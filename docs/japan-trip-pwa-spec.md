# Technical Specification — Japan Family Trip PWA

## 1. Project Overview

Build a Progressive Web App (PWA) that serves as an offline-first trip companion for a 12-day family trip to Japan (June 15–27, 2026: Memphis → Atlanta → Tokyo → Kyoto → Hiroshima (day trip) → Osaka → Tokyo → home). The app is the single source of truth for the day-by-day itinerary, hotel/booking details, train seat assignments, packing/pre-trip checklists, and a per-day budget tracker. It must work fully offline (subway, airplane mode, no data) and, when online, deep-link out to navigation/maps and taxi apps.

**Primary user:** Android phone (Chrome). **Secondary users:** 3 family members on iPhone (Safari), installed via "Add to Home Screen."

**Source material:** `/mnt/user-data/uploads/Japan_Family_Itinerary_2026.pdf` (28 pages). This PDF is the canonical content source — the coding agent must extract all itinerary content from it (see Section 4, Data Extraction).

## 2. Technical Decisions

| Decision | Choice | Rationale |
|---|---|---|
| App type | PWA (installable web app) | Single codebase works on Android (Chrome) and iOS (Safari "Add to Home Screen"); no app store submission needed; can ship in hours |
| Stack | Static HTML + vanilla JS + CSS (no framework, no build step) | Minimizes build complexity; trivially deployable to GitHub Pages; fast load on mobile data |
| Hosting | GitHub Pages | User already has a GitHub account/workflow (gaia-homelab repo); free, HTTPS by default (required for service workers) |
| Offline strategy | Service Worker with cache-first strategy for all app assets (HTML/CSS/JS/JSON/icons); itinerary data bundled as a local JSON file, not fetched from an API | Guarantees full offline availability with zero backend; no server costs or maintenance during the trip |
| Data storage (runtime) | `localStorage` for user-generated state: checklist check states, budget entries, "current day" override | Simple, synchronous, no dependencies; per-device (each family member's checklist/budget is independent — acceptable since this is a personal tool, not shared state) |
| Data source (itinerary) | Single static `itinerary.json` file generated by the coding agent from the PDF, bundled into the repo | Itinerary is fixed/known at build time; no need for a database or CMS |
| Navigation/maps integration | Deep-link via Google Maps Universal URL scheme: `https://www.google.com/maps/dir/?api=1&destination=<lat,lng or address>&travelmode=<mode>` | Universal links work cross-platform (open Google Maps app if installed, else web) without needing native app SDKs; no API key required for simple deep-links |
| Taxi integration | Deep-link to Google Maps with `travelmode=driving` as a baseline; optionally add a GO Taxi / Uber Japan link where the itinerary explicitly recommends a taxi | No taxi API integration — itinerary already specifies "GO Taxi" as the recommended app; just link out, don't integrate |
| Date/"Today" detection | Use device local date (`new Date()`) compared against trip date range (June 15–27, 2026) to auto-highlight/scroll to current day; fallback to a manual day-selector if outside range or for testing | Avoids timezone complexity by using device-local time, which is what the traveler experiences |
| Icons/manifest | Generate a simple 192x192 and 512x512 PNG icon (cherry blossom or Mt. Fuji theme, matching the PDF's 🌸 branding) + `manifest.json` | Required for "Add to Home Screen" to produce a proper app icon on both platforms |
| Browser storage caveat | This is a real installable PWA (not a Claude.ai artifact), so `localStorage` and Service Worker `caches` API are fully supported and required | N/A — noting explicitly since these APIs are restricted in other contexts |

## 4. Data Extraction from Source PDF

The coding agent must read `/mnt/user-data/uploads/Japan_Family_Itinerary_2026.pdf` and transcribe its content into `itinerary.json` following the schema below. Extract data **page by page** — the PDF has 28 pages covering: cover/legend (p.1), trip-at-a-glance (p.2), pre-trip checklist (p.3–4), and Day 0 through Day 12 itineraries (p.5–23), hotel recommendations (p.24), food guide (p.25–27), and practical tips (p.28).

**Important extraction notes:**
- The itinerary contains a dual-column "Option A / Option B" format on several days (Days 1, 9, 10, 11) and a "Schedule / Notes" two-column format on others. Flatten both into a single ordered list of activity blocks per day, preserving labels (e.g., mark Option B items so the UI can visually distinguish "suggested additions" from the "original plan" if desired — see optional `variant` field below).
- Preserve all addresses verbatim (including Japanese-language addresses) — these are needed for maps deep-links.
- Preserve all train/Shinkansen seat assignments and booking reference numbers exactly as written (these are critical reference info, e.g., booking reference `CPXU6R`, SAMURAI've booking ID `141053`, seat assignments for Days 5, 7, 9, 11).
- Where the PDF gives a named place without coordinates, the agent should geocode well-known places (e.g., "Senso-ji Temple", "Fushimi Inari Shrine") to lat/lng during data prep if feasible, OR simply use the place name/address as the `destination` query param for Google Maps (Google Maps accepts free-text addresses in the `destination` param — coordinates are not required). **Recommended: use address/place-name strings rather than attempting geocoding**, to avoid extra tooling.

### 4.2 Activity object fields

| Field | Type | Notes |
|---|---|---|
| `id` | string | Unique per activity, format `d{dayNumber}-{index}` |
| `time` | string\|null | As written in PDF (e.g. "10:30", "Evening") |
| `title` | string | Short name of activity/stop |
| `description` | string | Full details, tips, costs — markdown-lite allowed (line breaks as `\n`) |
| `variant` | "A"\|"B"\|null | "A" = original plan, "B" = suggested addition, null = single-option days |
| `location` | object\|null | `{ "name": "...", "address": "..." }` — used to build maps deep-link. `address` may be a Japanese address string or romanized place name; both work in Google Maps |
| `travelMode` | "walking"\|"transit"\|"driving"\|null | Used to pre-set the Google Maps `travelmode` param |
| `notes` | array of strings | Side-panel "tips" content associated with this activity (from the Notes column) |
| `bookingRef` | string\|null | Booking IDs / confirmation numbers (e.g. "Booking ID: 141053", "CPXU6R") |

## 5. Feature Specifications

### 5.1 Core: Itinerary Viewer (offline)

- Single-page app with a top-level day navigator: horizontal scrollable tab/pill list, "Day 0" through "Day 12", each showing date and short theme.
- Tapping a day shows that day's full itinerary as a vertical timeline of activity cards.
- Each activity card shows: time, title, description (collapsible if long), and any notes/tips in a visually distinct sub-block (e.g., lighter background, "💡 Tip" icon).
- For days with Option A/B columns (Days 1, 9, 10, 11): render as two visually labeled groups ("Original Plan" / "Suggested Additions") within that day, OR interleave chronologically with a small "A"/"B" badge on each card — **agent's choice, but must visually distinguish the two**.
- Hotel note banner at top of each day when relevant (e.g., "Still at Hotel Emion — no checkout today").
- "Day overview" footer per day: walking distance, transit ride count, budget estimate (from PDF).

### 5.2 "Jump to Today"

- On load, compare device local date to `trip.startDate`–`trip.endDate`.
- If within range: auto-select and scroll to the matching day.
- If before range: show Day 0, with a banner "Trip starts in N days".
- If after range: show Day 12 (or last day), with banner "Trip completed — viewing itinerary".
- Always show a persistent "Today" button in the header that jumps back to the auto-detected day regardless of which day is currently being viewed.

### 5.3 Maps / Navigation Deep-Links

- Every activity with a non-null `location` shows a "🧭 Navigate" button.
- Button constructs and opens: `https://www.google.com/maps/dir/?api=1&destination={encodeURIComponent(location.address || location.name)}&travelmode={travelMode || 'transit'}`
- Open via `window.open(url, '_blank')` — on Android Chrome this triggers the "Open in Google Maps app?" prompt if installed; on iOS Safari it opens Google Maps app via universal link if installed, else maps.google.com in browser. No native intent/URL-scheme handling needed beyond this.
- For activities where the PDF explicitly recommends a taxi (e.g., "Taxi is easiest late evening"), add a secondary small label/icon next to the Navigate button noting "🚕 Taxi recommended" (text only, no separate deep link needed — same Maps button with `travelmode=driving` covers ride-hailing navigation use case).

### 5.4 Pre-Trip Checklist

- Separate "Checklist" tab/view, accessible from main navigation.
- Renders `preTripChecklist` as grouped, collapsible sections (by category: "6-8 Weeks Before", "2-3 Weeks Before", "1 Week Before").
- Each item is a tappable checkbox/row. State persisted to `localStorage` under key `japan-trip-checklist-state` as `{ [itemId]: boolean }`.
- Show a progress indicator per category (e.g., "3/7 done") and overall.
- Include a "Reset all" button (with confirmation) for re-use on future trips.

### 5.5 Budget Tracker

- Separate "Budget" tab/view.
- Pre-populate with the per-day budget **estimates** from the PDF (`dayOverview.budgetEstimate`, given as yen ranges, e.g., "¥60,000–70,000 for family of 4") as read-only reference rows, one per day.
- Allow the user to add **actual spend entries**: simple form with `day` (dropdown, Day 0–12), `category` (free text or simple dropdown: Food / Transport / Shopping / Activities / Other), `amountYen` (number), `note` (optional text).
- Store entries in `localStorage` under key `japan-trip-budget-entries` as an array of `{ id, day, category, amountYen, note, timestamp }`.
- Show running total: total spent so far, and total spent per day, displayed alongside the estimate for that day for quick comparison.
- Display a simple converted USD estimate alongside yen totals using a **hardcoded approximate rate** (e.g., 1 USD = 155 JPY — agent should note this is a static approximation, not live FX, consistent with offline-first design). Label clearly as "approx."

### 5.6 Reference Data Views

- "Hotels" view: list of booked hotels (`bookedHotels`) with check-in/out dates and addresses (with Navigate buttons), plus the recommended-but-not-booked options (`hotels`) for reference, clearly separated.
- "Trains" view: list of all Shinkansen segments (`trains`) showing date/day, train name, car number, and a per-traveler seat table. This is pure reference data (no interaction needed) but should be easy to screenshot/show to staff.
- "Food Guide" view: the must-try foods list (`foodGuide`) and restaurant picks (`restaurantPicks`), grouped by city, as a browsable reference (could double as inspiration when deciding where to eat).
- "Tips" view: practical tips (`practicalTips`) grouped by category (weather, money, connectivity, etc.) as simple collapsible sections.

### 5.7 Navigation Structure

Bottom navigation bar (mobile-friendly, 5 tabs, persistent across views):

1. **Itinerary** (default/home) — day-by-day viewer (5.1–5.3)
2. **Checklist** (5.4)
3. **Budget** (5.5)
4. **Info** — combined Hotels + Trains view (5.6)
5. **Food & Tips** — combined Food Guide + Practical Tips (5.6)

## 6. PWA Configuration

### 6.1 `manifest.json`

```json
{
  "name": "Japan Family Trip 2026",
  "short_name": "Japan Trip",
  "start_url": "./index.html",
  "scope": "./",
  "display": "standalone",
  "background_color": "#ffffff",
  "theme_color": "#e85d75",
  "icons": [
    { "src": "icons/icon-192.png", "sizes": "192x192", "type": "image/png" },
    { "src": "icons/icon-512.png", "sizes": "512x512", "type": "image/png", "purpose": "any maskable" }
  ]
}
```

- `theme_color`: pick a color matching the cherry-blossom (🌸) theme from the source PDF (e.g., a sakura pink `#e85d75` or a deep red/torii-gate red `#cc4b4b`) — agent's discretion.
- Link manifest in `index.html` `<head>`: `<link rel="manifest" href="manifest.json">`.
- For iOS Safari, also add: `<meta name="apple-mobile-web-app-capable" content="yes">`, `<meta name="apple-mobile-web-app-status-bar-style" content="default">`, `<link rel="apple-touch-icon" href="icons/icon-192.png">`.

### 6.2 `service-worker.js`

- Cache-first strategy. On `install`, cache all static assets (`index.html`, `style.css`, `app.js`, `itinerary.json`, `manifest.json`, icons).
- On `fetch`, serve from cache first, fall back to network, and update cache on successful network fetch (stale-while-revalidate is acceptable but cache-first is simpler and sufficient since content is static).
- Register service worker in `app.js`:
```javascript
if ('serviceWorker' in navigator) {
  window.addEventListener('load', () => {
    navigator.serviceWorker.register('./service-worker.js');
  });
}
```
- Bump a `CACHE_VERSION` constant and delete old caches on `activate` so future content updates (e.g., correcting an address) propagate after redeploy.

### 6.3 GitHub Pages Deployment (document in README.md)

1. Create a new GitHub repo (e.g., `japan-trip-2026`), push all files to `main` branch.
2. Repo Settings → Pages → Source: `Deploy from a branch` → `main` / root.
3. Site will be available at `https://<username>.github.io/japan-trip-2026/`.
4. **HTTPS is automatic on GitHub Pages**, which is required for service worker registration.
5. On each phone: open the GitHub Pages URL in Chrome (Android) or Safari (iOS) → use browser menu → "Add to Home Screen" / "Install app".
6. After install, the app works fully offline (confirm by enabling airplane mode and relaunching from the home screen icon).

## 7. Visual Design Guidance

- Mobile-first, single-column layout, large tap targets (min 44px height for buttons/checkboxes).
- Use the emoji already present in the source material (🌸⛩️🗼🐙🚅🍜 etc.) as lightweight iconography — no icon library needed.
- Color palette suggestion: sakura pink/red accent (`#e85d75` or similar) for headers/active states, neutral light background, dark text for readability outdoors/in sunlight.
- "Option B" (suggested additions) badges/sections should use a visually distinct accent (e.g., green, matching the 🟢 from the source PDF) vs "Option A" (🔵 blue).
- Keep typography legible at arm's length: base font size ≥ 16px.

---

## 8. Out of Scope (explicitly not required)

- User accounts, multi-device sync, or shared/collaborative state across family members' phones.
- Live flight/train status integration.
- Live currency conversion (use hardcoded approximate rate per Section 5.5).
- Push notifications.
- Editing itinerary content through the UI (itinerary.json is static; corrections require redeploying).
